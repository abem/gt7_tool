## 1. 結論（仮説と判定）

**`current_laptime`（decoder.py:370、パケットオフセット 0x80）の正体は「ゲーム内の時刻（time of day）の進行 [ms]」である。ラップ経過時間ではない。**

- レースの時刻設定が「固定」の場合は**定数**、「時間進行あり」の場合はレース設定の**倍率で加速しながら単調増加**し、**レース再スタートで開始時刻に巻き戻る**。ラップ境界では一切リセットされない。
- 近傍オフセットのフィールド（0x74 lap_count / 0x76 total_laps / 0x78 best_laptime / 0x7C last_laptime）はすべて実データで妥当であり、**誤っているのは 0x80 のラベル1件のみ**。
- ライブ側への実害: **あり**（§3。走行中ラップタイム表示・ライブデルタ・推定ラップの3系統が実走行時に無意味な値になる）。ただし例外・クラッシュは発生しない「静かな誤動作」であり、TEST MODE では露見しない（§3.3）。

### 根拠3: 順当なラップ完了時に一度もリセットされない（全数分類・65回）

標本中の lap_count 変化は**延べ65回**。これを「順当な+1ラップ完了」（lap_count が1以上から+1に増える＝真のラップ境界）と「非連続遷移」（-1⇔0 の出入り・0起点・複数飛躍・逆行＝メニュー/リトライ/再スタート等の状態遷移）に**全数分類**した:

| 分類 | 件数 | うち current_laptime 減少（リセット） |
|------|------|--------------------------------------|
| 順当な+1ラップ完了 | **23回** | **0件** |
| 非連続遷移（-1⇔0・飛躍・逆行） | 42回 | 5件 |
| 合計 | 65回 | 5件 |

真の「現在ラップタイム」なら順当なラップ完了23回で**毎回リセット**されるはずだが、**0件**。減少5件はすべて非連続遷移側で発生しており、遷移内容と減少先の値は次のとおり（scan_result_v2.json より全件）:

```
2026-07-11_03_22_24: lap -1→0  clt 61,515,716 → 61,200,000 (=17:00:00.000 ちょうど)
2026-07-11_03_22_24: lap  0→4  clt 61,202,769 → 61,200,000 (=17:00:00.000 ちょうど)
2026-07-11_03_22_24: lap -1→0  clt 61,515,699 → 61,200,000 (=17:00:00.000 ちょうど)
2026-07-12_00_03_48: lap  5→9  clt 43,669,048 → 43,200,000 (=12:00:00.000 ちょうど)
2026-07-14_03_24_18: lap -1→0  clt 58,200,000 → 51,300,000 (=14:15:00.000 ちょうど)
```

5件全てが**きりの良い時刻ちょうどへの復帰**であり、「レース再スタート／別イベント選択でそのレースの開始時刻に戻る」挙動そのもの。ラップタイマーのリセット（→0付近）は一度も観測されていない。なお§2冒頭の挙動分類表で「非単調」とした6ファイル中、上記以外の減少（例: 66,612,250→65,700,000=18:15:00.000）は lap_count 変化を伴わない時点で発生した同種の巻き戻しである。

**分類の定義（明示）**: 「順当な+1」は `lap_count が1以上 かつ +1 増加` のみ。0→1（レース開始＝完了ラップなし）・-1⇔0（非走行状態の出入り）・飛躍・逆行は非連続遷移に分類した。初版レポートが記載した「52回・0件」はこの分類を行う前の誤集計であり、一次データ（65回・5件）と不一致だった。本節の数値は是正走査 v2 の出力（scan_result_v2.json）と1対1で一致する。

### 根拠4: 値域が1日以内に収まる

観測された全値が 0〜86,400,000ms（24時間）の範囲内。

### 根拠5: 近傍フィールドは妥当（0x80 だけの問題）

- last_laptime / best_laptime（0x7C/0x78）: 実測値は 95〜300秒級の ms でラップタイムとして妥当。ラップ切替時のみ更新される挙動も正しい
- lap_count / total_laps（0x74/0x76）: 整合
- よって decoder.py の抽出オフセットのずれではなく、**0x80 に付けた名前（意味づけ）の誤り**である

### 補: packet_def.json について

指示書§4の留意どおり、packet_def.json に 0x80 / current_laptime の記載はなく（「decoder.py が正」と自己記載する部分資料）、一次証拠にならない。本診断は外部資料に依存せず**実データの数値的整合性のみで自立的に確定**した。

## 3. ライブ側への実害判定

### 3.1 実害あり（実走行時に無意味な値を表示）

| 箇所 | 使用 | 症状 |
|------|------|------|
| websocket.js:418-419 | `RUNNING`（走行中ラップタイム表示） | 「時刻」を mm:ss.fff 整形するため、例えば 14:15 固定レースでは常に `855:00.000` を表示。進行レースでは倍率分速く進む「謎のタイマー」になる |
| telemetry-analysis.js:247（t軸）→ 505（liveDelta） | ライブデルタ（#lap-delta・デルタバー、DRIVEの大型デルタにもミラー） | 基準ラップの t[] も現在の t も「時刻」のため、デルタ＝「基準ラップ通過時からの時刻経過」となり**常に巨大な正値**（分〜時間オーダー）。バーは常時+側飽和（=常に「遅い」表示）。時刻固定レースでは t が全点同値となり距離補間の意味も失う |
| telemetry-analysis.js 538-574 | 推定ラップ（#est-lap-time） | `refLap.totalTime + liveDelta` の liveDelta が壊れているため無意味な推定値。`.pb` 判定も機能しない |

### 3.3 TEST MODE で露見しない理由（=これまで未発見だった理由）

- test-mode.js:435 はデモデータとして `current_laptime: posInLap * stepMs`（ラップ内経過ms・毎ラップ0リセット）を**正しい意味で合成**している。よって TEST MODE ではデルタ・推定ラップ・RUNNING がすべて完全動作する。
- 検証は TEST MODE 中心（PS5 不要のヘッドレス検証体制）であり、かつ実走行中のダッシュボード閲覧が殆どない（GT7改修計画書§1.3: 走行時刻と閲覧時刻は概ね不一致）ため、実走行でのみ現れる本症状は観測機会自体がなかった。
- 記録データ（gt7data）にも「時刻」がそのまま保存されており、P1-3 の実データ検証で初めて数値の不合理（14時間相当のラップタイム）として顕在化した。

## 4. 対応方針案（実装はしない。提案のみ）

| 案 | 内容 | 影響範囲 | リスク/工数 |
|----|------|---------|------------|
| **案1（推奨）: フロントのみ是正** | telemetry-analysis.js の時間軸を「受信実時刻のクランプ付き累積秒」（P1-3 REVIEW で実証済みの方式）に変更し、websocket.js の RUNNING 表示を「ラップ開始検知からの実時間経過」に変更。decoder.py・バックエンド・記録スキーマは無変更 | フロント2ファイル | 低〜中 / 0.5〜1人日。TEST MODE＋実データ（REVIEW比較）で検証可能。時刻進行倍率の影響も自動的に消える |
| 案2: decoder 正名化（スキーマ v3） | 0x80 を `day_time_ms` 等の正名で追加し、`current_laptime` は廃止または「ラップ内経過時間の導出値」に置換 | decoder.py・記録スキーマ・フロント・REVIEW API と広範 | 中〜高 / 記録データの世代がさらに増える。意味の正しさでは本命だが、B案ローテーションで旧世代データが世代交代する時期に合わせた方が移行コストが小さい |
| 案3: 該当表示の撤去 | RUNNING・ライブデルタ・推定ラップを撤去 | フロント | 低いが機能喪失。P1の目的（見る価値の提供）に逆行するため非推奨 |

**推奨**: まず案1（実走行でのデルタ系の即時復旧・低リスク）。案2 は将来のスキーマ整理（B案運用開始後）に統合して再提案するのが費用対効果に優る。なお現行の RUNNING 表示等の症状は**放置してもクラッシュ・データ破損には至らない**ため、緊急度は「中」（機能価値の毀損のみ）。

### 補足: 副次的な発見（対応不要・記録のみ）

- 記録データには「lap_count はあるが current_laptime が無い」中間世代（2026-02-11〜02-12）が存在する。P1-3 実装はフィールド単位のフォールバックで既に吸収済み。

## 2. データ供給設計

### 2.1 API: **既存 `GET /api/laps/{file}` をそのまま使用（バックエンド変更なし）**

- 必要な機能（`fields` 射影・`every` 間引き・to_thread化・Cache-Control）は P1-3 実装済みのAPIが全て備える。**新設エンドポイント・API変更は不要**と判断する。
- 要求フィールド: 計画書§2の突合表に基づく**58種**（全JSの `data.*` 消費フィールドの機械抽出結果）を `fields=` に明示指定する。定数 `REPLAY_FIELDS` として replay-mode.js に保持（コメントに再抽出コマンドを明記し、カード追加時の追随手順を残す）。
- 除外最適化: `course`（コース名は初回フレームのみ必要）等の毎フレーム不要なものも v2 記録では各サンプルに存在するが、**58種のまま取得する**（射影の複雑化より単純性を優先。実測17.2MB/典型ラップは許容済み）。

### 2.2 2段ロード（采決裁#1）

1. **先行**: `every=6`（10Hz、典型2.9MB実測）を取得 → 取得完了後すぐ再生・スクラブ可能にする。
2. **背景**: `every=1`（60Hz、典型17.2MB実測）を非同期取得。完了時に**一時停止せずシームレス差替**する:
   - 差替はバッファ交換＋「現在再生位置の時間値→新バッファの索引」再マップで行う（時間軸は両バッファとも同じ timestamp 起点のため写像は単調・一意）。
   - 差替中のフレーム供給は旧バッファで継続し、次フレーム要求から新バッファを参照（アトミック切替）。
3. 60Hz取得が失敗した場合は10Hzのまま継続し、ステータス表示（§4.4）に「10Hz再生」と明示する（黙って劣化しない）。

### 2.3 フレームバッファと索引

- バッファ: 取得サンプル配列そのまま（オブジェクト配列）。
- **時間索引**: 各フレームの経過秒 `t[i]` = timestamp のクランプ付き累積（dt<2s のみ加算。#128 と同一規則）。ギャップ（dt≥2s）は「中断点」としてマークし、スクラバー上に切れ目表示（§4.3）。
- **距離索引**: `d[i]` = position x/z の弦長積算（120mクランプ。REVIEW/telemetry-analysis と同一規則）。距離→フレームの逆引きは d[] の単調非減少性を利用した二分探索。
- 再生範囲: `lap_count>=1` の有効区間のみ（#128 の収集ゲートと同じ判定）。先頭のメニュー/グリッド区間はスキップ。

## 3. 多段フォールバック（采決裁#3・初期実装に含める）

### 3.1 サイズ事前判定（追加通信なし）

- 一覧API（`/api/laps`）が返す `size_bytes`（生ファイルサイズ）から射影後サイズを推定する: **推定射影サイズ ≒ size_bytes × 0.89**（実測係数: 典型ファイル 17.2/19.3=0.89、最大ファイル 151.06/169.7=0.89 の2点で一致確認済み）。
- 判定は再生ボタン押下時にクライアントで即時実施（HEAD等の追加リクエスト不要）。

### 3.2 閾値と段階（具体値。運用で調整できるよう replay-mode.js 冒頭の定数群に集約）

| 段階 | 条件（生 size_bytes） | 挙動 |
|------|----------------------|------|
| 通常 | ≦ 34MB（射影60Hz推定 ≦ 30MB） | 2段ロード（10Hz→60Hz） |
| 低レート強制 | 34MB超 〜 101MB（射影60Hz推定 30〜90MB） | **60Hz差替を行わず** 30Hz（every=2）を上限とする。ステータスに「30Hz再生(大容量)」表示 |
| 区間分割 | 101MB超（射影60Hz推定 > 90MB。全数走査で6件が該当、最大169.7MB） | 中断点（§2.3）で区切った**有効区間の一覧を先に提示**し、選択区間のみ 10Hz→30Hz でロード（1区間は通常ラップ相当まで縮小） |
| 縮退 | 区間分割でも単一区間が101MB超相当、または取得失敗 | 再生不可の明示表示＋REVIEW（距離チャート比較）への案内 |

- 根拠: 閾値30MB（射影）は「LAN数秒ロード・parse後~90MBヒープ」の上限目安。90MB超はparse後~270MB級となり通常端末でも危険（#132差し戻しで確定した全数実測: 最大151MB射影→450MB級ヒープ）。
- 区間分割の実装: サーバ変更なしで行うため、区間切り出しはクライアント側で `every=6` 先行データの中断点解析→本ロードは全量取得ではなく**必要区間の再生時にのみ60Hz化を断念し30Hz上限**とする（60Hz区間ロードにはAPIの範囲取得が必要になるため、初期実装はここまで。API拡張=range パラメータ追加は将来課題として§8に明記）。

## 5. 排他・状態管理設計

### 5.1 ライブ受信との排他（采決裁#7: 受信・記録は継続、表示のみ排他）

- 新規トップレベル `let replayActive = false`（replay-mode.js。衝突チェック要領は従来どおり全ファイルgrep）。
- **追加分岐は2箇所**（testModeActive の全参照の機械洗い出しに基づく全数。査差し戻しにより1箇所目のみだった初版を是正）:
  1. **websocket.js:616** `processTelemetryFrame()` の既存 testModeActive ゲートに `replayActive` を併記。現行「TEST MODE中はライブフレームを捨てる（接続維持・受信継続）」→「TEST MODE中**または再生中**は捨てる」。WS接続・受信・サーバ側記録（gt7data書込）は完全に無影響。
  2. **steer-response.js:80** `updateSteerResponse()` 内の**車固有中立操舵ゲイン（`STEER_RESP.neutralL`）自動較正ガード**に `replayActive` を併記: `var inTest = (typeof testModeActive !== 'undefined' && testModeActive) || (typeof replayActive !== 'undefined' && replayActive);`。理由: 再生フレームも単一入口経由で updateSteerResponse に到達するが、較正は「ライブ接続中の車の実測」を前提とした**永続状態（resetAnalysis/resetLapState の対象外）**であり、過去セッション（別車種含む）の再生データで汚染すると再生終了後も残存する。TEST MODE が同一リスクを較正除外で回避している設計（実装コメント「実走基準を汚さないため」）と対称に扱う。
- websocket.js の接続ピル保護ガード2箇所（:565/:584）は変更不要（§4.4 のとおり接続表示はライブのまま）。

再生フレームが到達する全更新関数（計画書#132§2の突合表）について、モジュールレベル可変状態を機械洗い出し（`^let/^var/^const .*= {` 全数grep＋書込点確認）し、再生汚染の観点で分類した:

### 5.2 モード間排他マトリクス（設計確定）

| 遷移 | 挙動 |
|------|------|
| REVIEW → 再生 | review-mode解除→replay-mode。REVIEWの選択状態は保持（終了時に復元表示） |
| 再生 → 終了 | §5.5 の復元列→ANALYSISライブ表示へ。直前がREVIEW起点なら REVIEW へ戻す |
| 再生中に TEST MODE 開始 | ツールバー操作で可能性あり → `startTestMode` は触らず、**replay側が testMode プロキシの MutationObserver で検知して再生を自動終了**（review-view.jsのDRIVE検知と同型・既存パターン） |
| 再生中に DRIVE/REVIEW 切替 | 同様に再生を自動終了してから遷移（Observer検知） |
| ライブ走行中に再生開始 | 可能（表示が再生に切替わるだけで受信・記録は継続）。ただし §4.4 のライブ接続表示で受信中であることは見える |

### 5.3 レースエンジニア通知の抑止（采決裁#4）

- telemetry-analysis.js の `checkEngineer`/`pushNotification` は**無改変**。抑止は供給側で行う: replay-mode.js がフレーム供給時に `analysisState.notif` を直接操作しない代わりに、**再生開始時に通知フラグを退避し、再生中は `analysisState.els.feed` を一時的に `display:none`（replay-mode CSS: `body.replay-mode .engineer-feed{display:none}`）**とする。トースト自体は発火してもDOM非表示・4秒で自動撤去されるため実害なし（CSSのみで抑止が完結し、JS無改変で済む）。
- 終了時: resetAnalysis（§5.5）が通知キュー・フラグを全初期化する（既存実装がこの用途を既にカバー。telemetry-analysis.js:133-145 実在確認済み）。

### 5.4 セッション集約スタットの再計算（采決裁#6）

- 対象: MAX SPEED（lapState.maxSpeed）、TOP SPEED通知系（notif.topSpeed）、一貫性σ（lapTimesMs）、ラップ履歴。
- 方式: **再生開始時に `resetLapState()`+`resetAnalysis()` を呼ぶ**（TEST MODE開始時と同じ扱い）。以降は再生フレームが既存関数経由で「そのラップ内の」値を自然に再構築する（MAX SPEED はシーク後方移動時のみ過大になり得るため、**シーク時に resetLapState→シーク位置までのラップ内最大を距離索引から再計算して直接セット**する。再計算は speed_kmh の区間max走査=O(n)ローカル計算）。

### 5.5 終了時の復元（既存パターン踏襲）

`stopTestMode`（test-mode.js:127-146 実在）と同一の復元列を replay 終了時に実行する:
1. 再生タイマ停止・バッファ解放（大容量配列の参照切り=GC対象化）
2. `resetAnalysis()`＋`resetLapState()`（偽BEST/履歴/通知の残留防止。TEST MODEで実証済みの既存関数）
3. `replayActive=false` → ライブフレームの表示が自然再開
4. body.replay-mode 解除・REVIEW起点なら review-mode 復帰

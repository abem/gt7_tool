# TEST MODE ガイド

## 概要

TEST MODEは、PS5がなくてもダッシュボードの動作を確認できる機能です。デモデータは「完全な合成テレメトリパケット」として組み立てられ、**実走の WebSocket 受信とまったく同一の処理経路（`handleTelemetryMessage`）** に流されます。このため TEST MODE はデモ表示であると同時に、**ライブ表示経路そのものの検証**になります（ライブでしか埋まらない項目が TEST MODE で空のまま、という乖離が起きません）。

## 使用方法

### 1. ダッシュボードを起動

```bash
docker compose up --build
```

### 2. ブラウザでダッシュボードを開く

```
https://localhost:8080
```

> 自己署名証明書を使用しているため、初回は「この接続ではプライバシーが保護されません」等の警告が出ます。Chrome は「詳細設定」→「localhost にアクセスする（安全ではありません）」で続行してください。

### 3. TEST MODEを有効化

ヘッダーのツールバーにある **● TEST MODE** トグルをクリックします（実行中はピルが点灯し、● ドットが点滅します）。誤操作防止のため、250ms 以内の連打は無視されます。

### 4. デモデータの表示を確認

合成パケットがライブと同じ経路で処理されるため、ほぼ全ての表示項目が埋まります：

- 速度、RPM、ギア位置、レブリミッター/TCS フラグ
- スロットル、ブレーキ、クラッチ入力（TCS/ABS 補正後の値も同値で供給）
- タイヤ温度、ホイール回転速度 (rps)、タイヤ半径
- サスペンション高さ、路面法線・路面距離、車体高さ (HGT)
- 座標 (X/Y/Z)、速度ベクトル、方角 (HDG)
- ラップ数、現在ラップ経過、LAST/BEST ラップ、ラップ履歴（周回ごとに 40ms ずつ短縮され PB 更新・通知も検証できる）
- 燃料残量・1周消費量・残り周回予測（1周目は未確定 `--` → 2周目から数値、という立ち上がり挙動もデモ）
- ギア比・変速最高速、車両 ID、スタート順位（グリッド順位 P3/16）
- 車体加速度（sway/heave/surge、m/s²）、加減速 G チャート
- STEER RESPONSE のデモ操舵（アンダー→ニュートラル→オーバーを周期的に巡回）

> **STEER RESPONSE だけはデモ専用駆動です。** ライブ経路の生舵角（±0.3rad 程度）では弧が小さすぎて見えないため、舵角を約6倍に誇張したデモ値で毎フレーム上書き描画されます。それ以外の全項目はライブと同一経路・同一変換で更新されます。

### 5. TEST MODEを停止

ツールバーの **● TEST MODE** をもう一度クリックすると、デモデータの生成が停止します（ピルの点灯と ● ドットの点滅が消えます）。停止時には、デモの偽値がライブ復帰後に残留しないよう、**ラップ系（ラップ数/LAST/BEST/履歴）・最高速・グリッド順位がリセット**されます。

## デモ軌跡データ

TEST MODEは、サーキットのレイアウトを模した20点の座標データ（`demoTrajectory`、`test-mode.js`）を内部に持っています。この20点を1周として合成パケットの座標・速度ベクトル・ラップタイムが生成され、座標表示（POS/VEL）やラップ解析ロジックを一巡させます（コース推定機能自体はTEST MODEでは使用されません）。

なお、セクタータイム（S1/S2/S3）は GT7 のテレメトリパケットにセクター情報が含まれないため、TEST MODE を含めダッシュボード全体で非対応です。

## デモデータ仕様

### 仕組み（合成パケット方式）

`renderDemoFrame()`（`test-mode.js`）が、`decoder.py` / `main.py` のキー名に忠実な合成テレメトリパケットを1フレームにつき1個組み立て、ライブ受信と同一の単一入口 `handleTelemetryMessage()` を1回呼びます。UI・解析・DRIVE ビュー・CAR ATTITUDE・チャート・STEER RESPONSE の全系統が、ライブ受信時と同じ経路・同じ変換で更新されます。

### 生成される主なデータ

| フィールド | 値 | 説明 |
|-----------|---------|------|
| position_x / position_z | -300～300 / -300～250 | デモ軌跡20点のコース座標 |
| velocity_x/y/z | 軌跡差分から導出 | 軌跡の向き × 現在速度 (m/s)。HDG 表示も駆動 |
| speed_kmh | 80～220 | コーナーごとに変化 |
| rpm / max_rpm | 0～6500 / 9000 | 周期変化 + ノイズ。レブ警告も検証可 |
| gear / gear_ratios | 1～6 / 6速分 | 10フレームごとにシフトアップ。ギア比表示も埋まる |
| throttle_pct / brake_pct / clutch | 0～100 / 0～50 / 0 or 1 | 補正後の値 (filtered) も同値で供給。クラッチはシフト時のみ踏む |
| tyre_temp / wheel_rps / tyre_radius | 50～95 / 10～15 / 0.33～0.34 | 4輪分 |
| susp_height | 30～70mm 相当 | decoder 契約どおり m 単位で供給（表示側で mm 換算） |
| road_plane_x/y/z / road_plane_distance | 微小変動 | 路面法線・路面距離 |
| current_laptime / last_laptime / best_laptime | 合成ラップ | 下記「合成ラップ」参照 |
| current_fuel / fuel_per_lap / fuel_laps_remaining | 漸減 | 1周目は未確定センチネル 0（表示は `--`）、2周目から数値 |
| body_accel_sway/heave/surge | m/s² | surge は速度波形の微分から導出（sway と単位を統一） |
| car_id / package_id | 1234 / 42 | 車両 ID 系 |
| pre_race_position / num_cars_pre_race | 3 / 16 | グリッド順位（P3/16 表示） |
| transmission_max_speed | 78 m/s | 変速最高速（表示側で km/h 換算） |
| flags | 動的 | rev_limiter / tcs_active 等が入力に連動 |

### 合成ラップ

`demoTrajectory` 20点 × 200ms間隔＝1周 4000ms を基準に、`last_laptime` は周回ごとに 40ms（`DEMO_LAP_SHORTEN_MS`）ずつ短縮されます。周ごとに PB を更新し続けるため、PB 判定・レースエンジニア通知・ラップ解析まで一巡検証できます。実車のラップタイムを模した値ではありません。

### 更新頻度

デモデータは**5Hz**（200ms間隔、`constants.js` の `TEST_MODE_CONFIG.intervalMs=200`）で更新されます。この間隔は UI/チャート/マップ/回転の全スロットル閾値より長いため、毎フレーム全系統が更新されます。

## TEST MODEの用途

### 開発時

- UI変更の視覚確認（APEX Broadcast リデザイン後の見た目確認）
- レスポンシブレイアウトのテスト
- **DRIVE / ANALYSIS ビューの切替確認**（ヘッダーのツールバー **ANALYSIS│DRIVE** セグメントで走行用最小表示と全パネル表示を切替）
- 新機能の動作確認（距離基準ラップ解析チャート等）

### デモ・プレゼンテーション

- ダッシュボードの機能紹介
- UIデザインの共有
- スクリーンショット撮影

### トラブルシューティング

- PS5接続問題の切り分け
- ダッシュボード単体の動作確認

## TEST MODEの制限

- 実際の車両データとは異なります（値はすべて合成）
- デモのラップタイム・BEST・最高速・グリッド順位は**停止時にリセット**され、ライブ復帰後には残留しません
- コース推定機能は使用されません（合成パケットにコース情報は含まれません）
- バックエンドのデータ保存機能（`gt7data/` への保存）は動作しません（ブラウザ内で生成されるためサーバーを経由しません）
- セクタータイム（S1/S2/S3）は GT7 がセクター情報を送信しないため、TEST MODE を含め非対応です

## PS5接続との関係（ライブ描画の一時停止）

**TEST MODE 実行中は、PS5 からの実走テレメトリを受信してもライブ描画は一時停止されます**（60Hz のライブと 5Hz のデモが表示を奪い合うのを防ぐため、WebSocket 取り込み側で遮断されます。再接続やデータ受信自体は継続します）。実走データに戻すには TEST MODE を停止してください。

接続ステータスピルは TEST MODE 中は `TEST MODE` 表示になり、停止時には接続状態に応じて `Connected`（テレメトリ処理済み）/ `Connected (no data)` / `Disconnected` に戻ります。

## 関連ファイル

- `index.html`: TEST MODE プロキシボタン（ツールバー導入後は非表示）のHTML構造
- `menu.js`: ヘッダーの操作ツールバー（**● TEST MODE** トグルがプロキシボタンを駆動）
- `test-mode.js`: テストモードのロジック（合成パケット生成 `renderDemoFrame`・開始・停止）
- `websocket.js`: ライブと共通の単一入口 `handleTelemetryMessage`（合成パケットの処理先）と TEST MODE 中のライブ取り込み遮断ガード
- `ui_components.js`: DOM要素キャッシュ（テストモードが使用）
- `steer-response.js`: STEER RESPONSE 描画（`updateSteerResponse` をテストモードのデモ操舵＝誇張舵角が上書き駆動）
- [docs/steer-response.md](steer-response.md): STEER RESPONSE の読み方・物理導出・較正の詳細
- `constants.js`: TEST MODE 設定（`TEST_MODE_CONFIG`: 更新間隔・ステータスピル文言）

## トラブルシューティング

### TEST MODEトグルが反応しない

- 直前のクリックから 250ms 以内の連打は無視されます（誤操作防止）。少し待ってからクリック
- ブラウザをリロード (F5)
- JavaScriptコンソールでエラーを確認 (F12)

### デモデータが表示されない

- コンソールログでエラーを確認
- ブラウザのコンソールで`[TEST]`タグのログを確認

> TEST MODE は合成パケットをブラウザ内で直接処理するため、WebSocket 接続（サーバー稼働）は必須ではありません。オフラインでも動作します。

### チャートが更新されない

- uPlotライブラリが正しく読み込まれているか確認
- ブラウザをリロード

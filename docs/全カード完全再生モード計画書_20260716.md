## 1. 目的とKPI

### 1.1 目的

**「過去ラップを選ぶと、走行中に見えていた全カード（約20枚）がそのまま動き出し、任意の瞬間へスクラバーで行き来できる」**状態にする。

- 誰が: ユーザー本人（走行後の振り返り）。
- どう使う: REVIEW（またはラップ一覧）から1本を選び「再生」→ ライブと同一のダッシュボード表示が記録データで駆動される。気になる区間（コーナー・挙動変化）へ時間/距離スクラバーで移動し、CAR ATTITUDEの姿勢・TYRES温度・PEDALS・STEER RESPONSE等を同時に観察する。
- 既存REVIEWとの棲み分け: REVIEW=2ラップの距離基準**比較**（俯瞰・定量）。本モード=1ラップの**全計器再現**（詳細・定性）。相互に補完し、REVIEWの一覧・選択UIを入口として再利用できる。

### 1.2 KPI（機械計測可能）

| KPI | 計測方法 | 目標（初期） |
|-----|---------|-------------|
| 再生モード利用回数 | 再生用フル取得（`GET /api/laps/{file}?every=1...`）のアクセスログ計数（既存 logging_middleware で計測可能・追加基盤不要） | 週1回以上 |
| 走行日のうち同日中に再生/レビューを使った日の割合 | 上記ログと gt7data 記録日の突合（改修計画書§2.2 KPIの拡張） | 50%以上 |

## 3. 技術方式（優先度つき・複数案）

### 案A（推奨）: 記録フレームを既存単一入口へ供給する「フロント再生コントローラ」

```
/api/laps/{file}?every=N&fields=<58種>  →  replay-mode.js（新規）
   フレームバッファ + 再生タイマ/スクラバー
        ↓ 1フレームずつ
   handleTelemetryMessage(frame, nowTs)   ← 既存・無改変の単一入口
        ↓
   全カード更新関数群（既存・無改変）
```

- **実証済みの土台が2つある（推測ではない）**:
  1. test-mode.js が「合成テレメトリ→handleTelemetryMessage→全カード駆動」を既に本番実装している（test-mode.js:465-467 の呼出し実測）。
  2. **#128の実データ再現ハーネスが「記録済み実フレーム→handleTelemetryMessage→カード群駆動」を既に実機で成立させた**（gt7data実ファイルを注入し RUNNING・デルタ・チャートが正しく駆動。WORK_20260716_162521/replay_*.json）。本機能はこのハーネスの製品化に相当する。
- 時間軸: フレームの `timestamp` 差分（クランプ累積）。#128でライブ解析系がこの方式に統一済みのため、再生でもそのまま整合する。
- スクラバー: 時間軸=フレーム索引（timestamp累積秒）。距離軸=REVIEWと同じ弦長積算（resampleByDist系の実証済みロジック）で「距離→フレーム索引」の逆引き表を作る。双方向に同期表示。
- 再生制御: 再生/一時停止/倍速（0.5/1/2/4x）/ステップ。タイマは timestamp 差分に倍率を掛けて駆動。
- データ取得: 既存 `/api/laps/{file}` が `fields`射影・`every`間引きを既にサポートしており**バックエンド変更は原則不要**。実測試算（§5・§8-1）に基づき「即応性: every=6(10Hz,2.9MB)で先行開始 → 背景で every=1(60Hz,17.2MB)へ差し替え」の2段ロードを推奨。

### 案B: サーバ側ストリーミング再生（WSで記録フレームを配信）

サーバに再生セッション管理・シーク・倍速制御を実装し、既存WS経路に流す。フロント変更は最小だが、**ライブ配信経路との混線リスク・バックエンド改修が大きく**、P1の「ライブ経路不変」原則に反しやすい。スクラバーのシーク応答もRTT依存。**非推奨**。

### 案C: REVIEWビューの拡張（チャートカーソル連動のみ）

REVIEWの距離チャートにカーソルを付け、その位置の数値をいくつか表示する。工数最小だが「全カード再現」にならず指示の目的を満たさない。**非推奨**（ただし案Aの部分縮退版として将来のフォールバックにはなり得る）。

## 4. 改修範囲

- コースデータベース・B案ローテーションへの変更

## 5. リスク・影響

| リスク | 内容 | 対策 |
|--------|------|------|
| ライブ受信との混線 | 再生中にWSライブフレームが届くと表示が交錯 | test-mode.js が同型問題を `testModeActive` ゲートで解決済み（websocket.js側にガード実在）。同じ方式で `replayActive` ゲートを追加（1条件分岐・既存パターン踏襲） |
| 共有グローバル状態の汚染 | lap-manager のBEST/履歴、analysisState、通知フラグ等が再生データで上書きされ、ライブ復帰後に残留 | stopTestMode の「状態リセット+復元」パターン（resetLapState/resetAnalysis 実在）をそのまま踏襲。再生終了時に全リセット |
| ペイロード規模（実測・**全数走査**） | 典型ラップ=6,784フレーム・生19.3MB。58フィールド射影で **60Hz:17.2MB / 30Hz:8.6MB / 10Hz:2.9MB**（実測値）。**全1,086件の真の最大は169.7MB（59,546フレーム・16.7分の混在セッション）で、84MB超が6件**（2026-07-08〜11に集中）。最大ファイルの射影は **60Hz:151.1MB / 30Hz:75.5MB / 10Hz:25.2MB**（実測）＝典型ケースの約8.8倍。なお全数中央値は1.8MB（短い記録が多数）で、「直近39件の中央値19.3MB」は最近の走行スタイル（長めの連続走行）を代表する値 | 2段ロード（10Hz先行→60Hz差替）を基本としつつ、**長大ファイルにはサイズ閾値ゲートを設ける**: 射影60Hz換算で閾値（例: 30MB、設計時に確定）超のファイルは (a)一律低レート（10Hz等）へ強制、または (b)ラップ/有効区間単位に分割しての部分再生のみ許可、または (c)再生非対応の縮退表示 — の多段フォールバック（採用の組合せは詳細設計で確定し§8-3と併せて采確認）。ロード進捗UI・中断可能設計は全ケース必須 |
| 長大・混在セッションファイル | 記録は「ラップ単位」だが実データにはメニュー滞在・再スタート混在の長尺ファイルが実在（#127診断で確認。**上記84MB超6件はいずれもこの型**で、最大169.7MBはlap_countが-1,0,1,2,5と非連続遷移する16.7分セッション） | 再生範囲はクロック/距離の有効区間に限定。スクラバーに中断ギャップを表示。上記サイズ閾値ゲートと併用。v1旧スキーマ（フィールド大幅欠落）は再生対象外とし縮退表示（§8-5） |
| UI到達性（P1の教訓） | 実データ規模（1,085件・84MB級）でのみ顕在化する不具合（P1-3実機不合格の再発防止） | 検証は本番同等全件の隔離環境で実施（P1-3是正で確立済みの手法）。性能計測（ロード時間・シーク応答・フレーム供給レート）を完了条件に含める |
| チャートのシーク再構築 | ストリップチャート（1200点リングバッファ）は逆方向シークで単純供給できない | シーク時は対象フレームから遡って窓分を再供給する方式（1200フレーム再計算・ローカル処理で軽量）を設計時に検証。困難時はシーク直後のチャートのみ空白から再蓄積（許容仕様として明記） |
| 視覚回帰ベースライン | 再生UI追加による差分 | REVIEWシナリオ追加と同じ手順でシナリオ追加+ベースライン更新（確立済み） |

## 7. 検証方針

1. **実データ再生検証**: #128の再現ハーネス（実フレーム→handleTelemetryMessage）を製品コード経由に置き換えて再実行し、代表カード群（速度・姿勢・タイヤ・デルタ・RUNNING）の値が記録と一致することを機械確認。v2複数世代・複数コースで実施。
2. **実データ規模検証**: 本番同等1,085件の隔離環境（P1-3是正で確立）で、一覧→再生開始→シークの到達性と性能（ロード時間・シーク応答<1s目安・60Hz供給の維持率）を計測。84MB級ファイルの縮退挙動を確認。
3. **排他・復元検証**: 再生⇔ライブ⇔TEST MODE⇔REVIEWの全遷移でJSエラー0・状態残留0（BEST/履歴/通知フラグ）。
4. **既存回帰**: 視覚回帰（再生シナリオ追加後 全PASS）・機能テスト10/10・既存スモーク維持・トップレベルシンボル無改変grep。
5. **実機検証**: abem2再ビルド→実ブラウザで実ラップ再生（査の実機検証フローに従う）。

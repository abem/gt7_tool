## 1. 結論

**原因**: `#review-root` は `.dashboard` の子要素だが、`.dashboard` は `grid-template-areas`（header/hero/left/center/right）による3カラムグリッド（styles.css:170-）であり、`#review-root` には **grid-area が未指定**。CSS Grid の自動配置により暗黙行の**第1カラム（`clamp(220px, 18vw, 272px)` ≈ 最大272px）に閉じ込められる**。その272px のコンテナ内で review.css の `grid-template-columns: 300px 1fr` が成立せず（サイドバー300px がコンテナ幅超過）、**第2カラム（比較サマリ・チャート群）が 0px に圧潰**。「1文字ごとの縦折り返し」はサマリ要素が幅0で折り返した見た目、「チャート完全空白」は幅0コンテナに ResizeObserver が uPlot を 0 サイズで追随させた二次影響。

**発生範囲（実測）**: **901px以上の全ウィンドウ幅**（証跡スクショの約1066pxに限らず、1920px・2560pxでも崩壊）。**900px以下では逆に正常**（review.css の狭幅メディアクエリで `.dashboard` 側も1カラム化するため偶然成立）。

**なぜ全検証をすり抜けたか**: 視覚回帰の `review_view` ベースライン（1920px）**そのものが崩れた状態で撮影されており**、以後の比較は「壊れた姿 vs 壊れた姿」の自己比較で常にPASS。スモーク（a3/a4/a5）は要素の存在・クラス・データのみ検査し**幅・ジオメトリを検査していない**。Playwright は幅0付近の要素もクリックできるため機能検証も通過。査のP1-3実機検証も本レイアウトの視覚確認を明示項目としていなかった。**＝視覚回帰は「変化しないこと」しか保証せず「正しいこと」は初回ベースラインの目視確認に依存する**という検証プロセスの穴（#135と同型の教訓）。

## 3. 修正案

### 案A（推奨・CSS 1行）

review.css に以下を追加:

```css
body.review-mode #review-root { grid-column: 1 / -1; }
```

（親グリッドの全カラムをスパンさせる。`grid-row` は自動のままで問題ない=ヘッダー行の下に配置される）

**有効性は実測で確認済み**（ブラウザ内動的注入・コード変更なし）: 全6幅で root がフル幅化し、main=706〜2200px・チャート実寸描画に回復（§2と同条件の after 計測値、スクショ layout_fixed_1066/1920.png）。副作用: 既存の狭幅（≤900px）は元々1カラムでスパン指定の影響なし。

### 同型の潜在バグ（併せて提案・適用スコープは計の判断）

`#replay-bar`（#133、現在は応急停止中）も同じ `.dashboard` 直下の grid-area 未指定子であり、**再生機能を復帰させると同型の圧潰が起こる**（未発現なのは応急停止中のため）。復帰前に同様の1行を replay.css へ:

```css
body.replay-mode #replay-bar { grid-column: 1 / -1; }
```

### 検証プロセスの是正（再発防止）

1. **視覚回帰ベースラインの新規取得時は人間/作の目視確認を必須化**（「ベースライン=正」の前提を初回に検証する。今回の根本的な検出漏れ点）
2. スモークに**ジオメトリassert**を追加（例: REVIEW表示時 `#review-main` 幅 > 400px、チャートコンテナ幅 > 300px）
3. 視覚回帰シナリオに**複数ウィンドウ幅**（900/1066/1400px等）の REVIEW/再生シナリオを追加

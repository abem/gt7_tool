### 1.2 現状構成（要約）

- **バックエンド**: Python 3ファイル（main.py=HTTP/WebSocketサーバ、telemetry.py=PS5 UDP通信、decoder.py=Salsa20復号）。受信テレメトリを `gt7data/` にラップ単位のJSONで自動記録（main.py:87-99）。
- **フロントエンド**: 単一ページ（index.html 551行）＋プレーン`<script>` 13本・計約4,800行。全JSが単一グローバルスコープを共有（ESモジュール不使用・ロード順依存）。約20カードを1画面に表示、DRIVE（走行用最小）/ANALYSIS（全表示）の2ビュー切替。WebGL依存は解消済み（CAR ATTITUDEはCanvas2D）。
- **デプロイ**: abem2上のDocker（`network_mode: host`、コードはイメージにCOPY → **変更反映には `docker compose up --build` が必須**）。HTTPS自己署名証明書で `https://192.168.1.129:8080`（閲覧はabem01等のLAN内ブラウザから）。
- **機能水準**: 機能テスト 10/10 PASS（verification_results.json）。リアルタイム表示・ラップ解析・STEER RESPONSE等、解析機能は多機能。

### 1.3 利用実態（ログ・データ実測。2026-07-16 15時時点）

**(a) 走行（データ受信）は現役で、ほぼ毎日発生している。**

- `gt7data/` に**1,085ラップJSON・計7.0GB**（du -sh 実測）。
- 記録日の分布（ファイル名日付の集計）: 2026-02-09〜02-13の4日間（133/287/402/73件）と、**2026-07-08〜07-16のほぼ毎日**（1〜57件/日）。**本日07-16も17ラップ記録**（12:48〜12:53 JST）。

**(b) 一方、ダッシュボードの閲覧はほぼ発生していない。**

- コンテナログ全期間（2026-07-11 21:51 UTC〜、約5日間）で、ページロード（`GET / -> 200`）は**計16回のみ**。うち07-16の3回は本チームの調査アクセスを含む可能性が高い。
- ページロード時刻と走行時刻（gt7dataの記録時刻）は概ね一致しない（例: 本日の走行は03:48-03:53 UTC、ページロードは04:14/05:52/06:01 UTC）。
- WebSocket接続には2h/4h/80hといった長時間セッションが常時1件程度存在する。自動再接続する「開きっぱなしのタブ」の可能性があるが、走行中に実際に画面表示されているかはログから判別不能（→ §8 未決事項）。

**(c) 記録データは「書きっぱなし」で、読み戻す機能がない。**

- `gt7data/` を読むのはコース学習用の `tests/test_course_detection.py --regenerate`（手動実行）のみ。**ダッシュボード本体に過去ラップを読み出す機能は存在しない**（main.pyは書き込みのみ）。
- ラップ履歴・ベストはセッション内のみで、ブラウザをリロードすると消える（USER_GUIDE.md FAQ「現在のセッション中のみ保存されます」）。
- 自動削除もなく、7.0GBが無期限に成長し続ける（USER_GUIDE.md FAQ「自動的には削除されません」）。

**(d) 新規導入者にとっては初回セットアップに致命傷候補を含む摩擦がある。**

- `.gitignore:84` で `ssl/` が除外されている一方、`Dockerfile:16` は `COPY ssl ./ssl` で ssl/ の存在を前提にしている。**証明書の生成手順・スクリプトはリポジトリ・README双方に存在しない**（grep該当0件）。フレッシュクローンでは ssl/ が無く、ビルドが失敗する可能性が高い（作者環境の手元証明書でのみ成立）。
- PS5 IPのデフォルト値が3系統で不一致（config.json:2=192.168.1.128 / main.py:52=192.168.1.100 / docker-compose.yml:12=192.168.1.128）。自動発見機構はなく手入力必須。env優先・config.jsonフォールバックの二重系統はREADME自身が「ハマりどころ」として注意書きしている（README.md:57）。
- 初回アクセスは必ず自己署名証明書の警告→手動続行（common-issues.md）。`network_mode: host` 前提のためDocker Desktop（Mac/Windows）では実質動作しづらい。

### 1.4 課題の特定（使われない原因）

上記の実測に基づき、原因を次の通り特定する（重み順）:

| # | 原因 | 根拠 |
|---|------|------|
| 原因1 | **走行後に見る価値が提供されていない**。走行中はGT7画面に目が占有されるのに、走行後に開いても過去ラップは何も見られない（ライブ専用・履歴はリロードで消滅・7GBの記録は死蔵） | §1.3 (b)(c) |
| 原因2 | **走行中の利用は第2画面前提で、成立しているか不明**。DRIVEビュー等の走行中機能は、走行環境に常設表示がなければ使われようがない | §1.3 (b)、未決事項 |
| 原因3 | **（外部利用者向け）導入摩擦**。ssl/欠落によるビルド失敗候補・IP手入力・証明書警告により、初回体験の前に離脱し得る | §1.3 (d) |
| 原因4 | **運用衛生の欠如**。gt7dataの無制限成長（7.0GB）は放置するとホスト運用リスクになる | §1.3 (c) |

**総括**: 機能不足や品質不良ではなく、**「毎日発生している走行データを、価値（振り返り）に変換する経路が無い」**ことが最有力原因である。ツールはデータを取り続けているのに、ユーザーがそれを見る動機・手段・タイミングのいずれも用意されていない。

## 2. 改修目的とKPI

### 2.1 目的

**「ほぼ毎日の走行が、走行後に振り返れるデータ資産になり、走りの改善につながる」状態にする。**
（誰が: ユーザー本人。どう使う: 走行後にPC/タブレットでその日のラップを開き、ベストとの比較・傾向を確認する。加えて、走行中の第2画面表示が環境的に成立するならDRIVEビューの利用ハードルを下げる。）

### 2.2 KPI（サーバーログ・gt7dataから機械計測可能なものを選定）

| KPI | 現状（実測） | 目標 |
|-----|-------------|------|
| 走行日のうち、同日中にダッシュボード閲覧があった日の割合 | ほぼ0%（直近5日: 走行5日に対し実閲覧ほぼなし） | 50%以上 |
| 過去ラップの読み出し（レビュー機能の利用）回数 | 0（機能自体が無い） | 週1回以上 |
| gt7data のディスク使用量 | 7.0GB・無制限成長 | 上限値内で安定（値は§8で確認） |

計測方法: `GET /` / WS接続ログ / （新設する）レビューAPIのアクセスログと、gt7dataの記録日を突合する。追加の計測基盤は不要。

## 3. 改修範囲

### 対象

- `gt7_tool/` 配下のバックエンド（読み出し系の追加）・フロントエンド・ドキュメント・運用スクリプト

- ライブ受信経路（telemetry.py/decoder.pyの受信・復号ロジック）の変更 — 現に安定稼働しており、触る理由がない
- abem01（閲覧側PC）やPS5側の環境構築・変更
- コースデータベースの再学習・拡充
- 既存UIの全面リデザイン（施策Eとして候補提示のみ。実施判断は効果測定後）

## 7. 検証・ロールバック方針

### 検証

1. **ヘッドレス・スモーク（PS5不要）**: TEST MODE＋Playwrightでダッシュボードの主要表示・新設REVIEWビューを自動確認（既存の検証手法を踏襲。JSエラー0件・機能テスト10/10維持を合格条件とする）。
2. **機械検査**: 既存 tests/ の回帰テスト一式、視覚回帰（再取得ベースライン基準で4/4 PASS）。
3. **実機検証**: abem2で `docker compose up --build -d` → abem01のブラウザで https://192.168.1.129:8080 を実閲覧し、(a)ライブ表示が従来どおり動くこと（走行またはTEST MODE）、(b)過去ラップのレビューが実データ（gt7data/の実JSON）で機能することを確認。
4. **B案（削除系）の追加検証**: dry-run出力と実削除対象の突合、削除前BUの実在確認。

### ロールバック

- コードはGit管理（**Git操作は計ロール専任**）。改修は機能単位のコミットに分割し、問題発生時は該当コミットのrevert → `docker compose up --build -d` で旧版に復帰する（イメージCOPY方式のため、リポジトリを戻して再ビルドすれば環境が旧状態に一致する）。
- gt7data/ はコード復帰の影響を受けない（volumeマウント）。B案による削除だけは不可逆のため、上記BU＋dry-run既定で防護する。
- 設定（.env/config.json）は変更前にBUを取得する（共通運用ルール準拠）。

### 1.1 対象コードの現状（実測。2026-07-16）

- バックエンド: main.py のルートは `GET /`（index）・`GET /ws`（WebSocket）・`GET /{filename}`（静的、.js/.css 許可リスト方式）の3本のみ（main.py:469-471）。gt7data/ への書込は save_lap_to_file（main.py:96-105）のみで、読み出す経路はダッシュボードに存在しない。
- gt7data/: 1,085ファイル・7.41GB（実測）。ファイル名形式 `YYYY-MM-DD_HH_MM_SS_CAR-{car_id}_Lap-{n}.json`。中身はサンプル（テレメトリ1フレーム=1オブジェクト）の配列。
- **スキーマが2世代ある**（実測）:
  - 旧（2026-02系、895ファイル）: 23キー。`lap_count` / `current_laptime` / `last_laptime` / `best_laptime` / `course` / `accel_g` が**無い**。`timestamp` / `position_x/z` / `speed_kmh` / `throttle_pct` / `brake_pct` はある。

- フロント: telemetry-analysis.js が距離基準解析の既存実装を持つ。純関数 `resampleByDist(samples, step)`（10m格子リサンプル）・定数 `STEP=10` / `DISCONTINUITY_M=120` はトップレベル共有スコープにあり、REVIEW ビューから**読み取り参照のみ**で再利用できる（改変・再宣言はしない）。

## 2. A案 詳細設計 — セッションレビュー機能

### 2.1 読み出しAPI仕様（バックエンド: main.py への追加）

追加はハンドラ2本とルート2行のみ。既存3ルートより**前**に登録する（`/{filename}` のワイルドカードに吸われないため）。

#### 2.1.1 `GET /api/laps` — ラップ一覧

- **データ源**: gt7data/ のディレクトリ走査＋ファイル名パースのみ。**ファイル内容は読まない**（1,000件超・7GBでも軽量。os.scandir で size/mtime を同時取得）。
- **ファイル名パース**: `^(\d{4}-\d{2}-\d{2})_(\d{2})_(\d{2})_(\d{2})_CAR-(\d+)_Lap-(\d+)\.json$`。不一致ファイル（バックアップ等）は一覧から除外する。
- **クエリパラメータ**:

| パラメータ | 型/範囲 | 既定 | 意味 |
|-----------|---------|------|------|
| `limit` | int 1〜1000 | 200 | 返却件数上限 |
| `offset` | int ≥0 | 0 | ページング |
| `date` | YYYY-MM-DD | なし | 記録日で絞り込み |
| `car_id` | int | なし | 車種IDで絞り込み |

- **並び順**: recorded_at 降順固定（新しい順）。
- **レスポンス** `200 application/json`:

```json
{
  "total": 1085,
  "laps": [
    {
      "file": "2026-07-16_03_52_08_CAR-3343_Lap-9.json",
      "recorded_at": "2026-07-16T03:52:08",
      "car_id": 3343,
      "lap_number": 9,
      "size_bytes": 21430118
    }
  ]
}
```

- **エラー**: 不正クエリ（非整数・範囲外）= 400。gt7data/ 不存在 = `{"total":0,"laps":[]}`（500 にしない。初回起動直後の正常状態）。

#### 2.1.2 `GET /api/laps/{file}` — 単一ラップ取得

- **ファイル名検証**（static_handler と同じ許可リスト思想）: 上記正規表現に完全一致した場合のみ受理。パス区切り・`..`・別拡張子は正規表現の時点で排除され 404。検証済みファイル名を `gt7data/` 直下とだけ結合する。
- **クエリパラメータ**:

| パラメータ | 型/範囲 | 既定 | 意味 |
|-----------|---------|------|------|
| `fields` | CSVフィールド名 | 既定セット（下記） | サンプルから返すキーの射影 |
| `every` | int 1〜60 | 6 | サンプル間引き（Nフレームに1つ。60Hz記録→every=6で約10Hz） |

- **fields 既定セット**（REVIEW ビューの距離基準比較に必要な最小集合）: `timestamp, current_laptime, speed_kmh, throttle_pct, brake_pct, position_x, position_z, gear, lap_count, last_laptime`。
  - 旧スキーマに無いキーは**そのサンプルに含めず返す**（欠落キーはクライアント側でフォールバック。§2.2.5）。未知のフィールド名指定はエラーにせず単に無視（400にしない。前方互換）。
- **レスポンス** `200 application/json`:

```json
{
  "meta": {
    "file": "2026-07-16_03_52_08_CAR-3343_Lap-9.json",
    "recorded_at": "2026-07-16T03:52:08",
    "car_id": 3343,
    "lap_number": 9,
    "samples_total": 29687,
    "samples_returned": 4948,
    "every": 6,
    "schema": "v2",
    "course": {"id": "tokyo_expressway_east", "name_ja": "東京インター（東ルート）"},
    "laptime_ms_approx": 102345
  },
  "samples": [ { "current_laptime": 1234, "speed_kmh": 182.3, "...": "..." } ]
}
```

  - `schema`: `lap_count` キーの有無で `"v2"`（2026-07系）/ `"v1"`（2026-02系）を判定して付す。
  - `course`: 先頭サンプルの `course` から id / name_ja / name_en のみ抽出（v1 は null）。
  - `laptime_ms_approx`: 最終サンプルの `current_laptime`（v1 は null）。ラップ確定値ではなく**近似値**であることを名前で明示（正確な確定値は次ラップの `last_laptime` にしか無く、単一ファイルからは得られないため）。
  - **【訂正 2026-07-16（実装時の事前検証で判明・計承認済み）】** decoder.py が `current_laptime` に格納する値（パケットオフセット 0x80）は実際には**ゲーム内時刻の進行(ms)**でありラップ経過時間ではない（実データで機械確認: 2026-07-15ファイル=全16,228サンプルで定数51,300,000 / 2026-07-16ファイル=58,200,000起点の単調増加）。よって `laptime_ms_approx` の算出は上記を廃し、**受信 `timestamp` のクランプ付き差分合計**（サンプル間 dt < 2s のみ加算。2s 以上は記録中断=メニュー放置・一時停止とみなし除外）に変更する。v1/v2 共通に機能し、実測で妥当性確認済み（例: 単純な先頭↔最終差 84,195s に対しクランプ後 268.9s ≒ サンプル数/60Hz）。decoder.py・ライブ経路は無変更（ライブ側への影響は Redmine #127 で計が別途追跡）。
- **性能設計**: 実測最大級のファイル（21MB・29,687サンプル）でも、既定 `every=6`＋既定 fields 射影で応答は概算 1.5〜2MB。`json.load` とフィルタ処理は `asyncio.to_thread` で実行し、イベントループ（=ライブWS配信）を阻塞しない（save_lap_to_file の書込スレッド化と同じ既存パターン、main.py:294）。
- **エラー**: 正規表現不一致/実体なし = 404。不正クエリ = 400。JSONパース失敗（破損ファイル）= 500 に `{"error":"corrupt file"}`。
- **キャッシュ**: `Cache-Control: no-cache`（既存ハンドラと同方針）。過去ラップは不変データだが、ローテーション（B案）でファイルが消え得るため保守的に合わせる。

#### 2.1.3 変更しないこと（明示）

- `/ws` の配信内容・頻度・接続管理（websocket_clients）には一切触れない。
- telemetry_background_task・save_lap_to_file のロジック不変（B案の記録ON/OFF分岐のみ P1-4 で追加。§3.5）。
- 静的配信 static_handler の許可リスト（.js/.css）は不変。gt7data/ へは新設 API 経由でのみ到達可能。

#### 2.2.4 距離基準比較ロジック（フロント計算）

ライブ解析（telemetry-analysis.js）と同じアルゴリズムを、**取得済みサンプル列に対して**適用する:

1. 各ラップのサンプル列から距離索引を構築: `position_x/z` の弦長積算。1フレーム弦長 > `DISCONTINUITY_M`(120m) はテレポート（ピット/リスポーン）としてスキップ（updateDistanceIndex と同一規則。every 間引き後は弦長が最大 every 倍になるため、判定閾値は `DISCONTINUITY_M * every / 6` 相当ではなく **`DISCONTINUITY_M` を every=6 の実効フレームレートで換算した 120m のまま**とし、実装時に実データ3件以上でスキップ誤発火が無いことを確認する）。
2. 時間軸: v2 は `current_laptime`/1000 [s]。v1（current_laptime 無し）は `timestamp`（ISO文字列）の先頭サンプルからの経過秒でフォールバック。
   **【訂正 2026-07-16（計承認済み）】** `current_laptime` は実はゲーム内時刻進行でありラップ経過時間ではない（§2.1.2 訂正注記参照）。時間軸は **v1/v2 共通で `timestamp` のクランプ付き累積経過秒**（サンプル間 dt < 2s のみ加算）に統一する。旧記載の「v2 は current_laptime」は廃止。
3. `{dist, t, speed, throttle, brake}` 列を組み立て、既存純関数 `resampleByDist(samples, STEP)` で 10m 格子へリサンプル（**関数は読み取り利用のみ。改変・再宣言しない**）。
4. デルタ: `delta[k] = A.time[k] − B.time[k]`（正=Aが遅い）。x軸は距離[m]、短い方の N でクリップ。
5. チャートは既存同梱の uPlot で新規インスタンス3個（速度/デルタ/入力）。ライブチャートのインスタンス・データ配列（speedData 等）には触れない。再描画は選択変更時のみ（毎フレーム処理なし。REVIEW はライブフレームに一切フックしない）。

## 3. B案 詳細設計 — gt7data 保存ポリシー

### 3.1 実測に基づく現状（2026-07-16）

| 項目 | 実測値 |
|------|--------|
| 総量 | 7.41GB / 1,085ファイル |
| 走行日あたり生成量（2026-07の9走行日） | 平均 635MB/日・最大 1,963MB/日（07-11） |
| ホストディスク（abem2 `/`） | 213GB 中 空き 65GB（使用70%） |
| ファイル所有者 | root（Docker コンテナが書込） |

無対策の場合、現ペース（毎日走行・平均635MB）で **約3.4ヶ月で空き容量を使い切る**計算（65GB ÷ 0.635GB/日 ≒ 102日）。ホストには他用途もあるため、実際の猶予はさらに短い。

### 3.2 方式: ローテーションスクリプト（新規 `scripts/gt7data_rotate.py`）

- スタンドアロン・標準ライブラリのみ（os/json/argparse/logging/datetime）。コンテナ不要（gt7data/ は volume マウントでありホストから直接見える）。
- **削除は2段階（trash方式）**: 対象ファイルは即時 unlink せず `gt7data_trash/YYYYMMDD/` へ **rename（同一ファイルシステム内move）**する。これが「削除前BU」に相当する（7GB級のコピーBUは物理的に高コストかつディスク圧迫が本末転倒のため、rename方式を採用）。trash 内で `trash_days` 経過したファイルのみ物理削除する。
- **選定ロジック**（優先順）:
  1. 保護リスト `gt7data/.rotate_keep`（1行1ファイル名）に載るファイルは常に対象外（ベストラップの手動ピン留め用）。
  2. 年齢基準: recorded_at（ファイル名から解釈。解釈不能ファイルは**触らない**）が `max_age_days` を超えるものを対象。
  3. 容量基準: 残り総量が `max_total_gb` を超えている間、古い順に対象へ追加。
- **dry-run 既定**: 引数なし実行は対象一覧・削減見込み量の**表示のみ**。実際の rename/削除は `--apply` 指定時のみ。さらに `--apply` は対象が総ファイル数の50%を超える場合は中断する（誤設定による全損防止のセーフティ）。
- **ログ**: 実行ごとに `scripts/logs/rotate_YYYYMMDD_HHMMSS.log` へ対象全件（ファイル名・サイズ・理由=age/size）を記録。dry-run も記録する。
- **権限**: gt7data/ 配下は root 所有のため、実行は `sudo`（ホスト cron の root 実行を提案）。スクリプト自体は権限不足時に明確なエラーで停止する（部分実行しない）。

### 3.3 閾値の提案値と根拠（★采承認事項。本番適用はまだ行わない）

| 設定 | 提案値 | 根拠 |
|------|--------|------|
| `max_total_gb` | **20** | 実測平均 635MB/走行日 → 約31走行日分（毎日走っても約1ヶ月の履歴を保持）。最大日 1,963MB でも10日分を確保。ホストディスク213GBの約9%・現空き65GBの約31%（trash 滞留を含めても50%未満）に収まり、他用途を圧迫しない |
| `max_age_days` | **180** | 低頻度利用期でも半年分は振り返り可能。A案レビューの主用途（直近セッションの振り返り・ベスト比較）には十分。2026-02の旧スキーマデータ（1.7GB）は本値の適用で自然に世代交代する |
| `trash_days` | **14** | 誤削除に気づいて復旧できる猶予2週間。trash 滞留の一時超過は最大でも直近2週間分（平均ペースで約9GB）に収まる |
| 実行頻度 | 週1（cron 日曜深夜） | 平均ペースで週あたり約4.4GBの増分に対し、20GB 上限との比で十分な追随性。走行時間帯（実測: 日中〜夜）と重ならない |

代替案として `max_total_gb=30`（約47走行日分・ディスクの14%）も成立するが、A案の利用実態（KPI: 週1レビュー）が確認できるまでは保守的な 20GB を推奨する。**いずれの値も dry-run 運用で実対象を確認した上で采が最終決定する。**

### 3.4 設定化（config.json への追加キー）

```json
"data_retention": {
  "enabled": false,
  "max_total_gb": 20,
  "max_age_days": 180,
  "trash_days": 14
}
```

- `enabled: false` を既定とし、リポジトリに入る初期値では**何も削除されない**。rotate スクリプトは `enabled: false` かつ `--apply` 指定時は「設定で無効化されています」と表示して終了する（`--force` での上書きは設けない。有効化は設定変更=采承認の証跡を必ず経由させる）。
- 環境変数上書き（RETENTION_* 系）は導入**しない**。削除系設定が .env と config.json の2系統に分かれると事故リスクが上がるため、config.json に一本化する（PS5_IP の二重系統が既にハマりどころとして文書化されている教訓の適用）。

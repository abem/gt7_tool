## 1. 目的とKPI

### 1.1 目的

走行後の振り返り（REVIEW/全カード再生）に、実レース現場で定着している**評価図・集計系**の分析を追加し、「見る」から「**課題がどこにあるか分かる**」へ進める（#142の知見: 瞬時値系は既存カバー済み、新規価値は評価図・集計系に集中）。

### 1.2 KPI

| KPI | 計測方法 | 目標 |
|-----|---------|------|
| レビュー/再生の利用（既存KPIの継続） | 既存アクセスログ突合 | 走行日の同日閲覧50%以上を維持・向上 |
| 新分析カードの参照（再生/REVIEW滞在の深化） | 追加計測は行わず、采/ユーザーの定性フィードバックで評価（過剰計測を避ける） | 「振り返りで課題特定に使えた」の確認 |

## 2. 対象6項目の詳細設計

### M-1【P1】G-G ダイアグラム

- **データ**: 横軸=`body_accel_sway`（m/s²実測→G換算表示）、縦軸=`accel_g`/`accel_decel`（main.py付加、G単位）。#142§2-B判定○。
- **表示**: 散布図1枚（点=フレーム、直近点を強調色）。摩擦円ガイド（0.5G/1.0G/1.5G同心円）を背景描画。
- **表示先**: **再生モード**（選択ラップ全体を描画し、再生位置の点をハイライト）と **REVIEW比較**（A/Bを2色重畳）。ライブへは追加しない（§8-1）。
- **描画方式**: Canvas2D 自前描画（散布は uPlot 不向き。ACCEL(G)チャート＝drawAccelChart と同じ既存流儀）。

### M-2【P1】コーナーフェーズ別デルタ ＋ トレイルブレーキ重なり指標

- **データ**: REVIEW比較の距離索引済み A/B 系列（resampleByDist 出力）＋既存 `classifyZone`（brake/throttle/coast、telemetry-analysis.js 実装済み・読み取り再利用）。
- **算出**:
  - 各10mグリッドをフェーズ分類し、フェーズ毎に Δt（A−B）を集計 →「進入(brake)/惰行(coast)/立上り(throttle)で各何秒得失」の表。
  - コーナー単位の内訳: 既存 `detectPeaksValleys` の valleys（速度極小=コーナー）を区間錨点に、コーナー毎の得失Top5リスト。
  - トレイル重なり指標: コーナー進入区間の `brake_pct × |wheel_rotation|` の正規化積分を A/B で比較（値が大きい=ブレーキを残して回頭。#142§2-Aの算出方法どおり）。
- **表示先**: REVIEW右ペインに新カード（サマリ帯の下）。数表＋コーナー別バー。
- **v1縮退**: v1（時間軸なし系列でも timestamp 経過で成立=P1-3方式）はフェーズ分類のみ可、トレイル指標は `wheel_rotation` 欠落のため「—」表示。

### M-3【P2】サスペンション変位ヒストグラム

- **データ**: `susp_height`[4]（60Hz実測）。#142§2-B判定○。
- **算出**: 選択ラップ（再生バッファ/REVIEW詳細）の4輪別度数分布（ビン幅は実データレンジから自動、20ビン目安）。
- **表示先**: 再生モードの新カード（4輪2×2の小ヒストグラム）。REVIEWはA/B重畳（半透明2色）。Canvas2D。

### M-4【P2】タイヤデグ率 ＋ ピットウィンドウ表示

- **データ**（**査差し戻し是正 2026-07-17**: 取得経路を実装実態に合わせ再設計）:
  - ラップタイム履歴: 既存の**永続グローバル** `lapState.lapTimes` / `analysisState.lapTimesMs`（実在確認済み）を読み取り。
  - 燃料系（`fuel_per_lap`・`fuel_laps_remaining`）と残周回（`lap_count`/`total_laps`）: **既存の永続グローバルにはキャッシュされていない**（フレーム毎に DOM へ書き捨て。websocket.js:267/280 実測）。よって race-metrics.js は **既存カードの DOM 表示値を1Hzで読み取る**（`elements.fuelPerLap.textContent` 等のパース。読み取り専用・ライブ経路JS無改変の原則を維持）。
  - DOM読取の結合リスクと対策: 表示フォーマット変更でパースが壊れ得るため、①パース失敗時は「--」縮退（黙って誤値を出さない）②スモークに「DOM値とパース結果の一致assert」を追加し、フォーマット変更時に検証で検出できるようにする。
  - 代替案（採否は査読時判断）: websocket.js の updateFuelState に「最新値を共有オブジェクトへ格納する1行」を追加すればDOM結合は消えるが、**ライブ経路JSに1行の変更が生じる**。本計画は原則（無改変）を優先し DOM 読取を主案とする。
- **算出**: デグ率 = 直近Nラップ（N=5目安）のラップタイム線形回帰勾配 [s/lap]（外れ値=ピットイン/ミス周は中央値ベースで除外）。ピットウィンドウ = 燃料切れ周（既存 fuel_laps_remaining）と残周回の交差から「L○○までにピット」を提示。
- **表示先**: **本項目のみライブ（ANALYSIS）に新ミニカード**（戦略情報はライブでこそ価値があるため）。ただし§3の方式により**既存JSは完全無改変**で実現する。
- **注意**: `current_laptime` は使わない（#127。ラップ確定値=last_laptime のみ使用）。

### M-5【P3】スロットル/ステア滑らかさスコア

- **データ**: `throttle_pct`・`wheel_rotation` の系列（再生バッファ/REVIEW詳細）。
- **算出**: 距離窓（100m目安）毎の一階差分の分散と符号反転回数（修正舵カウント）を 0-100 スコアへ正規化。**正規化の係数は実データでキャリブレーション**（実装工程で複数ラップの分布から決定し、根拠を実装時文書に記録。#142の裏取り原則を実装にも適用）。
- **表示先**: REVIEW比較サマリへの2値追加（A/Bスコア）＋再生バーの近傍にミニ表示。

### M-6【P3】energy_recovery・torque_vector の表示

- **データ**: `energy_recovery`（スカラ）・`torque_vector`[4]（v2実データに実在・現在未表示）。
- **表示**: 再生モードの新ミニカード（回生バー＋4輪トルク配分の十字バー）。
- **縮退**: 非対応車種（全編ゼロ/欠落）のラップでは「N/A（この車種はデータなし）」を明示（黙って空にしない）。

### 3.1 実装の器: 新規ファイル集約 + 既存ファイルへの追加は注入点のみ

### 3.2 REVIEW/全カード再生との関係

- REVIEW = **比較文脈**: M-2（フェーズ別デルタ）を主役に、M-1/M-3/M-5をA/B重畳で提供。
- 再生 = **単一ラップ文脈**: M-1（再生位置ハイライト付きG-G）・M-3・M-6を提供。再生のシーク/レート差替とはバッファ確定イベントのみで連動（#135の教訓: シーク時の部分再供給はせず、確定バッファから都度全量再計算。計算は10m格子ベースで軽量）。
- データ取得は両機能の**取得済みフレームを読み取り再利用**（新規fetchなし＝API・サーバ負荷の追加なし）。

## 7. 検証方針

1. **算出の正しさ**: 各指標について実データ2ラップ以上で独立検算（レポート#127/#132で確立した「本文数値と一次データの機械照合」方式）。
2. **実データ規模**: 本番同等1,085件環境で新カード表示・性能（描画<100ms目安/カード）。84MB級・v1・非対応車種の縮退を含む。
3. **視覚証跡＋ジオメトリassert**: 新カード全てにスクリーンショット証跡と min_widths assert。ベースライン新規分は取得時に目視確認（確認者記録）。
4. **全操作経路**: REVIEW⇔再生⇔ライブ⇔TEST MODE の全遷移×新カードの表示/クリア。再生のシーク・レート差替・区間切替後の再計算整合。
5. **既存回帰**: 視覚回帰全シナリオ・機能テスト10/10・既存スモーク・シンボル無改変grep・既存グローバルへの書込0件grep。
